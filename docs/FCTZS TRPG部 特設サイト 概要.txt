1. サイトの目的・概要
当サイトは、FCTZS TRPG部のメンバーがより快適で充実したTRPGライフを送るために開発された専用のポータルサイトです。
Discordのチャットだけでは過去に流れて埋もれてしまう「キャラクターシート」「通過したシナリオの履歴」「メンバーのスケジュール・プレイスタイル」「セッションの募集」などを一元管理し、ストック型の情報共有を行うことを目的としています。

2. 主な機能（ユーザー向け）
サイト内は大きく分けて以下の機能で構成されています。

🏠 ホーム（ダッシュボード）
未ログイン: 直近の予定・進行中セッションの概要とログイン誘導。
ログイン後: 自分の次回予定・自分の募集・予定カレンダー、その下に部全体の直近予定と進行中セッション。

👤 プレイヤー（メンバー）管理
プロフィール＆プレイスタイル: 各メンバーの「化身欲」「活躍欲」「物語欲」などを可視化するレーダーチャート表示。
推し設定: お気に入りの自陣キャラクターや「最強シナリオ」をピン留め。
GM可能シナリオ: 自分がGMできるシナリオを登録。他プレイヤーが「気になる」を押すと Discord DM で通知される。
部活外通過: サイト未登録のシナリオを個人の通過済履歴として登録（シナリオ一覧には出ない）。
スケジュール一括入力: 参加可能な日程（昼・夜）をカレンダー形式で「〇・△・×」で一括登録し、メンバー間で予定を合わせやすくする機能。

🎭 キャラクター管理
複数システム対応のステータス管理: クトゥルフ神話TRPG（6版/7版）、エモクロアTRPG、オリジナルシステム（ガイアケアTRPG）など。
代理作成対応: プレイヤー欄で所有者を選び、別メンバーのキャラを作成できる（編集権限は作成者の Auth に紐づく）。
ココフォリア一発出力: 能力値・技能からチャットパレットを生成してコピー。

📖 シナリオ・卓（セッション）管理
シナリオデータベース: 部内シナリオの一覧（検索・絞り込み）と詳細。
気になる: シナリオ詳細で興味をトグル。初回ON時のみ GM可能登録者へ Discord DM。
セッション記録: GM / PL / キャラクターの通過履歴を卓終了時に自動紐付け。
セッション一覧: キーワード・ステータス・システムで絞り込み。

📢 募集＆スケジュール管理
セッション募集: 新規セッションのPL/GM募集。Discord 募集チャンネルへ案内投稿。
カレンダー連動: 確定セッションとメンバーの空き予定を統合表示。観戦希望もセッション詳細から登録可能。

💬 なりきりチャット（BBS）
キャラクターロールプレイ: 自分が作成したキャラクターのアイコンと名前で発言。
なりすまし防止: Discord 認証により他人のキャラでの発言を防止。

3. システム構成（開発・技術者向け）
本サイトは、サーバーの維持費を極力抑えつつ、モダンで高速な動作を実現するために以下の技術スタックで構成されています。

フロントエンド（画面側）
言語: HTML / CSS / Vanilla JavaScript（ReactやVueなどの重いフレームワークは不使用）
ホスティング: Cloudflare Workers 静的配信（本番フロント例: https://fctzs.daruji.workers.dev/ ）
特徴:
  共通処理やAPI通信はすべて utils.js に集約。
  Chart.jsを用いたレーダーチャート描画機能。
  Supabase の認証情報（JWT）でログイン状態を判定。

バックエンド（API・定期処理）
環境: Cloudflare Workers (worker/index.js) — API 例: https://fctzs-trpg.daruji.workers.dev
役割:
  フロントエンドとデータベース（Supabase）の間に入るAPIゲートウェイ。
  Discord連携: 募集開始時、満員時、セッション当日リマインド、気になる通知（Bot DM）など。
  Cronジョブ: セッション当日通知や古い募集のクリーンアップなど。

データベース（DB・認証）
環境: Supabase (PostgreSQL)
役割:
  プレイヤー情報、キャラクター、シナリオ、予定などの全データ保存。
  RLS により書込は所有者単位。詳細スキーマは docs/DB_info.txt / DB-overview.md。
認証: Discord OAuth。players.user_id（Auth UUID）と players.discord_id（snowflake）を分離して解決。

4. 開発時の共通ルール（コーディング規約）
API通信の正規化: フロントエンドからは Utils.apiGet(), Utils.apiPost() などを経由して通信します。バックエンド（Worker）内では、Supabaseとの通信は共通ラッパー sbFetch() / sbServiceFetch() を通します。

権限管理（RLSの考慮）:
通常の読み取り・本人書込は利用者JWT。Discord Bot・Cron・代理作成など RLS を跨ぐ処理は Service Role（sbServiceFetch）を使い、呼び出し元の本人検証を Worker 側で行う。

UIのシンプル化:
リッチな表示が必要な場合（モーダルやツールチップなど）は、外部ライブラリに頼りすぎず、軽量なCSSとJSでコンパクトに実装することを推奨しています。
