1. サイトの目的・概要
当サイトは、FCTZS TRPG部のメンバーがより快適で充実したTRPGライフを送るために開発された専用のポータルサイトです。
Discordのチャットだけでは過去に流れて埋もれてしまう「キャラクターシート」「通過したシナリオの履歴」「メンバーのスケジュール・プレイスタイル」「セッションの募集」などを一元管理し、ストック型の情報共有を行うことを目的としています。

2. 主な機能（ユーザー向け）
サイト内は大きく分けて以下の機能で構成されています。

👤 プレイヤー（メンバー）管理
プロフィール＆プレイスタイル: 各メンバーの「化身欲」「活躍欲」「物語欲」などを可視化するレーダーチャート表示。

推し設定: お気に入りの自陣キャラクターや、面白かった「最強シナリオ」をピン留めしてアピール可能。

スケジュール一括入力: 参加可能な日程（昼・夜）をカレンダー形式で「〇・△・×」で一括登録し、メンバー間で予定を合わせやすくする機能。

🎭 キャラクター管理
複数システム対応のステータス管理: クトゥルフ神話TRPG（6版/7版）、エモクロアTRPG、オリジナルシステム（ガイアケアTRPG）など、システムに合わせた能力値や技能を登録可能。

ココフォリア一発出力機能: 登録したキャラクターの能力値、HP/MP（自動計算）、システムに準拠したチャットパレット（ダイスボット用コマンド）を生成し、ワンクリックでココフォリアにペーストできる状態でクリップボードにコピーする機能。

📖 シナリオ・卓（セッション）管理
シナリオデータベース: 部内で回ったシナリオの一覧と、システム、概要の管理。

セッション記録: 誰がGMを務め、誰がPLとしてどのキャラクターで参加したか（通過履歴）を自動的に紐付け。

📢 募集＆スケジュール管理
セッション募集: 新規セッションのPL募集をサイト上で行う機能。募集が立つと、自動的にTRPG部のDiscordサーバーへ通知（メンション付き）が送信されます。

カレンダー連動: 確定したセッションと、メンバーの空き予定を1つのカレンダー上で統合して表示。

💬 なりきりチャット（BBS）
キャラクターロールプレイ: 自分が作成したキャラクターのアイコンと名前を使って発言できる掲示板。

なりすまし防止: 裏側でDiscordアカウントとの連携（認証）を行っており、他人のキャラクターを勝手に使って発言できない安全な設計。

3. システム構成（開発・技術者向け）
本サイトは、サーバーの維持費を極力抑えつつ、モダンで高速な動作を実現するために以下の技術スタックで構成されています。

フロントエンド（画面側）
言語: HTML / CSS / Vanilla JavaScript（ReactやVueなどの重いフレームワークは不使用）

ホスティング: GitHub Pages (ronitoak.github.io)

特徴: * 共通処理やAPI通信はすべて utils.js に集約。

Chart.jsを用いたレーダーチャート描画機能。

Supabaseの認証情報（JWTトークン）をローカルストレージで管理し、各ページでログイン状態を判定。

バックエンド（API・定期処理）
環境: Cloudflare Workers (worker/index.js)

役割: * フロントエンドとデータベース（Supabase）の間に入るAPIゲートウェイ。

Discord連携: 募集開始時、満員時、セッション当日のリマインドなどをDiscordのWebhookやBot API経由で自動送信。

Cronジョブ（定期実行）: 毎日特定の時間に動作し、「翌日のセッションの自動リマインド通知」や「1ヶ月以上経過した古い募集データの自動クリーンアップ（削除）」を実行。

データベース（DB・認証）
環境: Supabase (PostgreSQL)

役割:

プレイヤー情報、キャラクター、シナリオ、予定などの全データ保存。

ON DELETE CASCADE 制約により、親データ（キャラクター等）が削除された際に紐づく技能やステータスも連動して安全に削除される設計。

認証: Discord OAuthを用いたソーシャルログイン。これにより「誰が操作しているか」を厳密に特定（なりきりチャット等で活用）。

4. 開発時の共通ルール（コーディング規約）
API通信の正規化: フロントエンドからは Utils.apiGet(), Utils.apiPost() などを経由して通信します。バックエンド（Worker）内では、Supabaseとの通信はすべて共通ラッパー関数である sbFetch() を通して行い、冗長なヘッダー設定などを省略しています。

権限管理（RLSの考慮）:
Supabaseのセキュリティルールにより、基本データは sbFetch(env, null, ...) を用いて環境変数の匿名キー（ANON_KEY）で読み書きを行いますが、なりきりチャット（postsテーブル）など厳密な本人確認が必要なエンドポイントでは sbFetch(env, request, ...) としてユーザーの証明書を引き継いで処理します。

UIのシンプル化:
リッチな表示が必要な場合（モーダルやツールチップなど）は、外部ライブラリに頼りすぎず、軽量なCSSとJSでコンパクトに実装（例：ココフォリア出力の高速ツールチップ等）することを推奨しています。
